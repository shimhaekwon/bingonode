## 1. 원인 분석

### 호출 경로
1. 버튼 클릭 → `changeTimeframe('W')` 등 — [stock-common.js](public/stock/js/stock-common.js)
2. `daysMap = { D:365, W:730, M:1825, Y:3650 }` → `days` 매핑
3. `POST /api/stock2/data { ticker, days }`
4. 백엔드 `stockFetcher.fetch(ticker, days)` → DB freshness 검사 → `getRange(ticker, days)`

### 근본 원인 — `stockFetcher.isDbFresh` 가 깊이(depth) 미검사

[libs/stockFetcher.js](libs/stockFetcher.js) 기존 `fetch`:
```js
if (this.isDbFresh(maxDate)) {                        // 최신성만 검사
    return await stockModel.getRange(ticker, periodDays);
}
```

- `isDbFresh`는 `maxDate >= 오늘`만 확인.
- DB가 요청 기간만큼의 **과거 데이터를 보유했는지(깊이)는 미검사**.
- `predict`가 모든 종목을 `fetch(ticker, **400**)`로 미리 채워두어 DB에 약 270 거래일치 적재 + `maxDate=오늘`.
- 그 결과: `days=3650`(년) 요청도 "fresh"로 판정 → `getRange(ticker, 3650)` → `LIMIT 3650` → 실제로는 보유분 ~270행만 반환.
- → 일/주/월/년 모든 요청이 같은 ~270행 반환 → **차트 불변**.

### 부수 사항 — 의미 모호성

`interval: '1d'` 고정 + `daysMap` 으로 일·주·월·년을 *조회 기간 길이*로 해석. 한국 증권사 관례인 *캔들 주기(일봉/주봉/월봉/년봉)*와 라벨 불일치.

## 3. 수정 설계 — 캔들 주기 지원

| 계층 | 변경 |
|---|---|
| 프론트 계약 | `{ticker, days}` → **`{ticker, period}`** (`D`/`W`/`M`/`Y`) |
| 백엔드 fetcher | Yahoo native interval 직접 사용 (`1d`/`1wk`/`1mo`) |
| 년봉(Y) | Yahoo가 yearly 미지원 → **월봉 받아 서버에서 연 단위 집계** |
| 캐시 | 인메모리 5분 TTL을 `ticker_period` 키로 분리 |
| 폴백 | Yahoo 실패 시 daily DB 에서 W/M/Y 집계 |

### INTERVAL_MAP

```js
const INTERVAL_MAP = {
  D: { yahoo: '1d',  yearsBack: 1  },
  W: { yahoo: '1wk', yearsBack: 5  },
  M: { yahoo: '1mo', yearsBack: 20 },
  Y: { yahoo: '1mo', yearsBack: 40, aggregate: 'year' },
};
```

### 집계 규칙 (월→년 / 일→W·M·Y)

| 필드 | 산출 |
|---|---|
| `open` | 버킷 첫 row의 open |
| `high` | 버킷 max(high) |
| `low` | 버킷 min(low) |
| `close` | 버킷 마지막 row의 close |
| `volume` | 버킷 sum(volume) |
| `date` | 버킷 시작일 (W=일요일, M=월 1일, Y=연 01-01) |
